The Data Bottleneck

  • Manual Entry: Enterprises process massive volumes of paper invoices and inventory logs by hand.
  • High Cost: Human transcription is time-consuming and expensive.
  • Error Prone: Manual data entry leads to inconsistencies in accounting and inventory systems.
  • Objective: Democratize IDP by transforming raw images into structured JSON automatically.

Technical Challenges

OCR Limitations

Variability in image quality (blur, skew) and non-standard layouts (curved wine labels) challenge traditional OCR engines.

The Semantic Gap

Raw text lacks structure. "TOTAL 125.50" is just a string. Converting this to {amount: 125.50} via Regex is brittle.

Architecture

Managing CORS policies and state for sequential batch processing in a split Frontend/Backend architecture.

Hybrid AI Pipeline

Combining EasyOCR for raw detection with Google Gemini for semantic understanding.

Microservices Architecture

A containerized, decoupled system designed for local deployment via Docker.

  • Frontend: Next.js 16 (App Router) with Radix UI.
  • Proxy Layer: Next.js API Route to handle CORS and API keys securely.
  • Backend: Python FastAPI serving EasyOCR and Gemini.
  • Processing: Asynchronous sequential batch processing to manage load.

The Extraction Pipeline

1. Optical Character Recognition

We utilize EasyOCR running on CPU (for portability). It handles the "Reading" phase, converting pixels into raw text strings. It supports French/English but is sensitive to noise.

2. Semantic LLM Processing

The raw text is fed into Gemini 2.5 Flash Lite. Using Few-Shot Prompting, the model classifies the doc (Invoice vs. Wine) and enforces a strict JSON schema output.

Key Engineering Hurdles

  • CORS & Security: Browser security blocked direct Backend calls. We implemented a Server-Side Proxy pattern in Next.js to forward requests, keeping API keys hidden.
  • Batch Processing: Parallel requests crashed the server. We built a Sequential Queue in the frontend using React Hooks to process one file at a time.
  • Strict Typing: transitioning raw LLM output to strictly typed TypeScript interfaces required robust error handling and parsing logic.
